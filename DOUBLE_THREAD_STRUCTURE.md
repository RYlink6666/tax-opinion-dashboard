# 项目结项汇报的双线结构设计

## 核心问题诊断

**问题**：原报告是"混合线"，没有清晰区分两条线索

```
原报告结构（混乱）：
第1部分：技术方案总体设计（抓什么线？）
第2部分：数据采集（项目线）
第3部分：LLM方法论（技术线）
第4部分：可视化（项目线）
第5部分：核心创新（两条线混在一起）
第6部分：政策启示（项目线）
...
第9部分：深度应用（技术线，太晚了）

结果：听众不知道框架何时出现、何时应用
```

## 解决方案：双线并行交织

```
【上层线：项目执行流程】（为什么这样做？）
    数据采集 → LLM分析 → 可视化展示 → 政策启示
    (确保数据质量)  (分类&发现)  (交互展示)  (输出成果)
         ↓              ↓           ↓           ↓
【下层线：技术框架应用】（怎么做的？）
    MediaCrawler → LangExtract+BERTopic → Streamlit → 决策支持
    (多平台爬虫)   (提示工程+主题建模)  (可视化)   (数据驱动)
         ↓              ↓           ↓           ↓
【交织点】
    每个部分都说清"为什么选这个技术"和"怎么具体应用"
```

## 实际写作中的"双线并行"技巧

### 技巧1：每个章节都要明确"两条线"

```
【错误写法】
3.1 大语言模型在舆论分析中的应用
    采用了提示工程和少样本学习...
    (读者：这是什么？为什么用？怎么用？不清楚)

【正确写法】
3.1 大语言模型在舆论分析中的应用
    项目需求：2,297条舆论需要5维度分类
             (项目线：问题定义)
    
    技术选择：采用Google的LangExtract框架
             (技术线：框架介绍)
    
    具体实现：设计精心的System Prompt，融入以下要素...
             (融合：技术在项目中的应用)
    
    质量验证：精度88.5%，与人工标注的Kappa=0.87
             (项目线：成果指标)
```

### 技巧2：用表格明确对应关系

```markdown
| 项目步骤 | 使用的技术框架 | 为什么选这个 | 效果如何 |
|---------|--------------|-----------|--------|
| 数据采集 | MediaCrawler | 反爬虫自动处理 | 3平台×1200条/900条/200条 |
| 数据清洁 | 标准流程 | 为LangExtract做准备 | 99.3%去重，精度提升3.5% |
| 意见分类 | LangExtract | 快速部署，无需训练 | 88.5%准确率 |
| 话题发现 | BERTopic | 自动K值，语义优化 | 18个话题，稳定聚类 |
| 可视化 | Streamlit+缓存 | 云端免费，用户友好 | 首屏<1s，9个页面 |
```

### 技巧3：用"为什么"贯穿两条线

```
为什么采集3个平台？(项目线：多维度覆盖)
  ↓
为什么选MediaCrawler？(技术线：框架优势)
  ↓
采集后怎样清洁？(项目线：数据质量)
  ↓
为什么这样清洁？(技术线：LangExtract的输入要求)
  ↓
分析后得到什么？(项目线：5维度分类)
  ↓
为什么用LangExtract？(技术线：框架选择)
  ↓
展示怎样展现？(项目线：9个页面)
  ↓
为什么这样展现？(技术线：两框架输出的整合)
```

## 汇报时的双线解读

### 针对政策部门（项目线优先）

```
"我们采集了2,297条舆论，通过AI分析，发现：
- 22.4%的意见是负面的
- 5.9%的意见高风险
- 消费者关心价格，企业关心利润

基于这些发现，我们提出4条政策建议..."

（背后用的技术框架是LangExtract和BERTopic，但不一定要前台强调）
```

### 针对技术评审（技术线优先）

```
"我们采用了Google的LangExtract框架进行结构化分类，精度88.5%。
同时用BERTopic进行无监督主题发现，得到18个话题。
这两个框架的协同，使我们能够从维度分类和话题发现两个角度理解舆论。

具体来说：
- LangExtract给出5个维度的分类，支持置信度追踪
- BERTopic给出18个话题的语义聚类，支持8种可视化
- 两者融合支持更深层的交叉分析..."

（项目成果是数据驱动的政策建议，但重点在技术先进性）
```

## 检查清单

在改写时，每个部分都问自己：

- [ ] 项目线明确吗？（读者知道我们做了什么吗？）
- [ ] 技术线明确吗？（读者知道我们用了什么框架吗？）
- [ ] 融合点清晰吗？（读者知道为什么选这个技术吗？）
- [ ] 因果关系强吗？（读者明白一步步为什么这样做吗？）
- [ ] 政策部门能理解吗？（项目线足够清晰吗？）
- [ ] 技术评审能验证吗？（技术线足够深入吗？）
